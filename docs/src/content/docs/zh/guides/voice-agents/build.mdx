<Aside type="caution">

请尽早确定你的架构：

- `OpenAIRealtimeWebRTC` 是最简单的浏览器方案，并且会为你处理音频输入/输出。
- `OpenAIRealtimeWebSocket` 提供更多控制能力，但你必须自己管理音频采集和播放。
- 函数工具会在 `RealtimeSession` 运行的位置执行。如果会话在浏览器中运行，工具也会在浏览器中运行。
- Realtime 交接会保持相同的语音和模型。如果你需要不同的后端模型，请通过工具进行委托，而不是使用交接。

</Aside>

## 会话设置

### 音频处理

某些传输层（例如默认的 `OpenAIRealtimeWebRTC`）会自动为你处理音频输入和输出。对于其他传输机制（例如 `OpenAIRealtimeWebSocket`），你需要自己处理会话音频：

当底层传输支持时，`session.muted` 会报告当前静音状态，`session.mute(true | false)` 用于切换麦克风采集。`OpenAIRealtimeWebSocket` 不实现静音：`session.muted` 返回 `null`，且 `session.mute()` 会抛出异常，因此在 websocket 配置下，你应在本地暂停采集，并在麦克风应重新开启前停止调用 `sendAudio()`。

### 会话配置

你可以在构造 [`RealtimeSession`](/openai-agents-js/openai/agents-realtime/classes/realtimesession/) 时，或调用 `connect(...)` 时传入额外选项来配置会话。

这些传输层允许你传入任何与 [session](https://platform.openai.com/docs/api-reference/realtime-client-events/session/update) 匹配的参数。

建议优先使用较新的配置结构，即 `outputModalities`、`audio.input` 和 `audio.output`。旧版顶层字段如 `modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription` 和 `turnDetection` 仍会为向后兼容而被标准化处理，但新代码应使用这里展示的嵌套 `audio` 结构。

对于较新的参数，如果在 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应参数，可使用 `providerData`。传入 `providerData` 的任何内容都会作为 `session` 对象的一部分直接传递。

可在构造时设置的其他 `RealtimeSession` 选项：

| Option                                        | Type                              | Purpose                                                              |
| --------------------------------------------- | --------------------------------- | -------------------------------------------------------------------- |
| `context`                                     | `TContext`                        | 合并到会话上下文中的额外本地上下文。                                 |
| `historyStoreAudio`                           | `boolean`                         | 在本地历史快照中存储音频数据（默认关闭）。                           |

| `outputGuardrailSettings`                     | `{ debounceTextLength?: number }` | 护栏触发节奏。默认 `100`；设为 `-1` 表示仅在完整文本可用时运行一次。 |
| `tracingDisabled`                             | `boolean`                         | 禁用会话追踪。                                                       |
| `groupId`                                     | `string`                          | 在多个会话或后端运行间对追踪进行分组。                               |
| `traceMetadata`                               | `Record<string, any>`             | 附加到会话追踪的自定义元数据。                                       |
| `workflowName`                                | `string`                          | 追踪工作流的友好名称。                                               |
| `automaticallyTriggerResponseForMcpToolCalls` | `boolean`                         | MCP 工具调用完成后自动触发模型响应（默认：`true`）。                 |
| `toolErrorFormatter`                          | `ToolErrorFormatter`              | 自定义返回给模型的工具审批拒绝消息。                                 |

`connect(...)` 选项：

| Option   | Type                                          | Purpose                              |
| -------- | --------------------------------------------- | ------------------------------------ |
| `apiKey` | `string \| (() => string \| Promise<string>)` | 此连接使用的 API key（或懒加载器）。 |
| `model`  | `OpenAIRealtimeModels \| string`              | 传输连接可选的模型覆盖。             |
| `url`    | `string`                                      | 可选的自定义 Realtime 端点 URL。     |
| `callId` | `string`                                      | 绑定到已存在的 SIP 发起通话/会话。   |

## 智能体能力

### 交接

与常规智能体类似，你可以使用交接将智能体拆分为多个智能体，并在它们之间进行智能体编排，以提升智能体性能并更好地界定问题范围。

<Code lang="typescript" code={multiAgentsExample} />

与常规智能体不同，Realtime Agents 中的交接行为略有不同。执行交接时，进行中的会话会更新为新的智能体配置。因此，智能体会自动访问当前对话历史，并且目前不会应用输入过滤器。

此外，这也意味着 `voice` 或 `model` 不能在交接中更改。你也只能连接到其他 Realtime Agents。例如如果需要使用不同模型（如推理模型 `gpt-5.2`），可以使用[通过工具委托](#delegation-through-tools)。

### 工具

与常规智能体一样，Realtime Agents 可以调用工具执行操作。Realtime 支持**函数工具**（本地执行）和**托管 MCP 工具**（由 Realtime API 远程执行）。你可以使用与常规智能体相同的 `tool()` 辅助方法来定义函数工具。

#### 函数工具

函数工具与 `RealtimeSession` 在同一环境中运行。这意味着如果会话运行在浏览器中，工具也会在浏览器中执行。如果需要执行敏感操作，请在工具内部调用后端，让服务器完成特权工作。

这样，浏览器侧工具就可以作为连接服务器端逻辑的轻量回传通道。例如，[`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) 在浏览器中定义了 `refundBackchannel` 工具，将请求和当前对话历史转发到服务器上的 `handleRefundRequest(...)`，在那里独立的 `Runner` 可以使用不同的智能体或模型评估退款，再将结果返回语音会话。

#### 托管 MCP 工具

托管 MCP 工具可通过 `hostedMcpTool` 配置，并在远程执行。当 MCP 工具可用性变化时，会话会发出 `mcp_tools_changed`。若要阻止会话在 MCP 工具调用完成后自动触发模型响应，请设置 `automaticallyTriggerResponseForMcpToolCalls: false`。

当前过滤后的 MCP 工具列表也可通过 `session.availableMcpTools` 获取。该属性与 `mcp_tools_changed` 事件都只反映活动智能体上已启用的托管 MCP 服务器，并且已应用智能体配置中的 `allowed_tools` 过滤。

#### 后台结果

工具执行期间，智能体将无法处理用户的新请求。一种提升体验的方式是让智能体在即将执行工具时先告知用户，或说一些固定短语来争取执行时间。

如果某个函数工具完成后不应立即触发新的模型响应，可从 `@openai/agents/realtime` 返回 `backgroundResult(output)`。
这会将工具输出回传到会话，同时将响应触发时机交由你控制。

#### 超时

函数工具超时选项（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）在 Realtime 会话中的行为相同。默认 `error_as_result` 下，超时消息会作为工具输出发送。使用 `raise_exception` 时，会话会发出包含 [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) 的 `error` 事件，并且该次调用不会发送工具输出。

### 护栏

护栏可用于监控智能体输出是否违反一组规则，并立即中断响应。这些护栏检查基于智能体响应的转写文本执行，因此要求启用模型文本输出（默认已启用）。

你提供的护栏会在模型响应返回时异步运行，使你可以基于预定义分类触发器中断响应，例如“提到某个特定禁用词”。

当护栏触发时，会话会发出 `guardrail_tripped` 事件。事件还会提供一个 `details` 对象，其中包含触发护栏的 `itemId`。

<Code lang="typescript" code={guardrailsExample} />

默认情况下，护栏每 100 个字符运行一次，并在最终转写可用时再运行一次。由于说出文本通常比生成转写更慢，这通常能让护栏在用户听到前就截断不安全输出。

如果你想修改此行为，可以向会话传入 `outputGuardrailSettings` 对象。

当你只想在响应结束时，对完整生成的转写评估一次时，设置 `debounceTextLength: -1`。

### 打断

使用内置语音活动检测时，用户在智能体说话时插话会自动触发智能体检测，并根据所说内容更新上下文。它还会发出 `audio_interrupted` 事件。这可用于立即停止所有音频播放（仅适用于 WebSocket 连接）。
