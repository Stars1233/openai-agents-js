## Runner 生命周期与配置

### 智能体循环

当您在 Runner 中使用 run 方法时，需要传入一个起始智能体和输入。输入可以是字符串（视为用户消息），也可以是输入项列表，即 OpenAI Responses API 中的各类条目。

随后 runner 会运行一个循环：

1. 使用当前输入调用当前智能体的模型。
2. 检查 LLM 响应。
   - **最终输出** → 返回。
   - **交接** → 切换到新智能体，保留累积的对话历史，回到第 1 步。
   - **工具调用** → 执行工具，将其结果追加到对话中，回到第 1 步。
3. 一旦达到 `maxTurns`，抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。

<Aside type="note">
  判断 LLM
  输出是否为“最终输出”的规则是：它生成了所需类型的文本输出，且没有工具调用。
</Aside>

#### Runner 生命周期

在应用启动时创建一个 `Runner`，并在请求间复用。该实例存储全局配置，如模型提供方和追踪选项。只有在需要完全不同的设置时才创建新的 `Runner`。对于简单脚本，也可以调用内部使用默认 runner 的 `run()`。

### 运行参数

传给 `run()` 方法的输入包括：启动运行的初始智能体、此次运行的输入，以及一组选项。

输入可以是字符串（视为用户消息）、[输入项](/openai-agents-js/openai/agents-core/type-aliases/agentinputitem)列表，或者在构建[人机协作](/openai-agents-js/zh/guides/human-in-the-loop)智能体时使用的 [`RunState`](/openai-agents-js/openai/agents-core/classes/runstate) 对象。

其他可选项包括：

| Option                  | Default | Description                                                                                                              |
| ----------------------- | ------- | ------------------------------------------------------------------------------------------------------------------------ |
| `stream`                | `false` | 若为 `true`，调用将返回 `StreamedRunResult` 并在模型事件到达时发出事件。                                                 |
| `context`               | –       | 转发给每个 tool / guardrail / handoff 的上下文对象。详见[上下文管理](/openai-agents-js/zh/guides/context)。              |
| `maxTurns`              | `10`    | 安全上限——达到后抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。     |
| `signal`                | –       | 用于取消的 `AbortSignal`。                                                                                               |
| `session`               | –       | 会话持久化实现。参见[会话](/openai-agents-js/zh/guides/sessions)。                                                       |
| `sessionInputCallback`  | –       | 会话历史与新输入的自定义合并逻辑；在模型调用前运行。参见[会话](/openai-agents-js/zh/guides/sessions)。                   |
| `callModelInputFilter`  | –       | 在调用模型前编辑模型输入（items + 可选 instructions）的钩子。参见[调用模型输入过滤器](#call-model-input-filter)。        |
| `toolErrorFormatter`    | –       | 自定义当工具调用被拒绝时返回给模型的消息。参见[工具错误格式化器](#tool-error-formatter)。                                |
| `reasoningItemIdPolicy` | –       | 控制在将先前运行条目回放为模型输入时，是否保留或省略推理条目的 `id`。参见[推理条目 ID 策略](#reasoning-item-id-policy)。 |
| `tracing`               | –       | 每次运行的追踪配置覆盖（例如导出 API key）。                                                                             |
| `errorHandlers`         | –       | 受支持的运行时错误处理器（当前为 `maxTurns`）。参见[错误处理器](#error-handlers)。                                       |
| `conversationId`        | –       | 复用服务器端会话（仅限 OpenAI Responses API + Conversations API）。                                                      |
| `previousResponseId`    | –       | 从上一次 Responses API 调用继续，而不创建会话（仅限 OpenAI Responses API）。                                             |

### 流式传输

流式传输允许您在 LLM 运行时接收额外的流事件。一旦开始流式传输，`StreamedRunResult` 将包含此次运行的完整信息，包括所有新生成的输出。您可以使用 `for await` 循环遍历这些流事件。更多内容见[流式传输](/openai-agents-js/zh/guides/streaming)。

### 运行配置

如果您要创建自己的 `Runner` 实例，可以传入 `RunConfig` 对象来配置 runner。

| Field                       | Type                     | Purpose                                                                      |
| --------------------------- | ------------------------ | ---------------------------------------------------------------------------- |
| `model`                     | `string \| Model`        | 为运行中的**所有**智能体强制指定模型。                                       |
| `modelProvider`             | `ModelProvider`          | 解析模型名称——默认为 OpenAI 提供方。                                         |
| `modelSettings`             | `ModelSettings`          | 覆盖每个智能体设置的全局调优参数。                                           |
| `handoffInputFilter`        | `HandoffInputFilter`     | 执行交接时变更输入项（若交接本身未定义）。                                   |
| `inputGuardrails`           | `InputGuardrail[]`       | 应用于*初始*用户输入的护栏。                                                 |
| `outputGuardrails`          | `OutputGuardrail[]`      | 应用于*最终*输出的护栏。                                                     |
| `tracingDisabled`           | `boolean`                | 完全禁用 OpenAI 追踪。                                                       |
| `traceIncludeSensitiveData` | `boolean`                | 在仍然发出 span 的同时，从追踪中排除 LLM/工具的输入与输出。                  |
| `workflowName`              | `string`                 | 显示在 Traces 控制台中——用于对相关运行进行分组。                             |
| `traceId` / `groupId`       | `string`                 | 手动指定 trace 或 group ID，而不是由 SDK 生成。                              |
| `traceMetadata`             | `Record<string, string>` | 附加到每个 span 的任意元数据。                                               |
| `tracing`                   | `TracingConfig`          | 每次运行的追踪覆盖（例如导出 API key）。                                     |
| `sessionInputCallback`      | `SessionInputCallback`   | 此 runner 上所有运行的默认历史合并策略。                                     |
| `callModelInputFilter`      | `CallModelInputFilter`   | 在每次模型调用前编辑模型输入的全局钩子。                                     |
| `toolErrorFormatter`        | `ToolErrorFormatter`     | 自定义当工具调用被拒绝时返回给模型消息的全局钩子。                           |
| `reasoningItemIdPolicy`     | `ReasoningItemIdPolicy`  | 在将生成的条目回放到后续模型调用时，关于保留或省略推理条目 `id` 的默认策略。 |

#### 服务器管理的会话

您可以让 OpenAI Responses API 为您持久化会话历史，而不是在每轮都发送完整的本地记录。这在协调长对话或多个服务时很有用。详情见[会话状态指南](https://platform.openai.com/docs/guides/conversation-state?api-mode=responses)。

OpenAI 提供两种复用服务器端状态的方式：

##### 1. 针对完整会话的 `conversationId`

您可以使用 [Conversations API](https://platform.openai.com/docs/api-reference/conversations/create) 创建一次会话，然后在每一轮复用其 ID。SDK 会自动仅包含新生成的条目。

## 钩子与自定义

### 调用模型输入过滤器

使用 `callModelInputFilter` 在调用模型*之前*编辑模型输入。该钩子会接收当前智能体、上下文以及合并后的输入项（包含会话历史时也包括在内）。返回更新后的 `input` 数组和可选的 `instructions`，以实现敏感数据脱敏、丢弃旧消息或注入额外的系统引导。

可在单次运行中设置（`runner.run(..., { callModelInputFilter })`），或在 `Runner` 配置中作为默认值（`RunConfig` 中的 `callModelInputFilter`）。

### 工具错误格式化器

使用 `toolErrorFormatter` 自定义当工具调用被拒绝时发送回模型的消息。这使您可以返回特定领域的措辞（例如合规指引），而不是使用 SDK 的默认消息。

该格式化器可在单次运行中设置（`runner.run(..., { toolErrorFormatter })`），或在 `RunConfig` 中全局设置（`new Runner(...)` 中的 `toolErrorFormatter`）。

该格式化器当前在 `approval_rejected` 事件时触发，并接收：

- `kind`（当前始终为 `'approval_rejected'`）
- `toolType`（`'function'`、`'computer'`、`'shell'` 或 `'apply_patch'`）
- `toolName`
- `callId`
- `defaultMessage`（SDK 的回退消息）
- `runContext`

返回字符串以覆盖该消息，返回 `undefined` 则保留 SDK 默认。如果格式化器抛错（或返回非字符串值），SDK 会记录警告并回退到默认的拒绝消息。

### 推理条目 ID 策略

使用 `reasoningItemIdPolicy` 控制当 SDK 将先前生成的运行条目转换回 `AgentInputItem[]` 以用于后续模型输入时，是否保留推理条目的 `id` 字段。

这会影响 SDK 将生成的模型条目作为输入回放的场景，例如：

- 同一轮中的后续模型调用（如工具执行之后），
- 后续轮次复用生成条目作为输入/历史，
- 从保存的 `RunState` 恢复的运行，
- 类似 `result.history` / `result.output` 的派生结果视图（这些是模型输入形状的数组）。

- `'preserve'`（默认）保留推理条目 ID。
- `'omit'` 在回传为输入前移除推理条目的 `id` 字段。
- 非推理条目不受影响。

以下内容**不**会改变：

- 原始模型响应（`result.rawResponses`），
- 运行条目（`result.newItems`），
- 提供方返回的当前轮次模型输出。

换言之，该策略适用于 SDK 由先前生成条目构建**下一次输入**时。

您可以在单次运行中设置（`runner.run(..., { reasoningItemIdPolicy: 'omit' })`），或作为 runner 默认值设置（`new Runner({ reasoningItemIdPolicy: 'omit', ... })`）。从保存的 `RunState` 恢复时，将复用先前解析的策略，除非您覆盖它。

#### 与 `callModelInputFilter` 的交互

`reasoningItemIdPolicy` 在 `callModelInputFilter` 之前应用。如果需要自定义行为，`callModelInputFilter` 仍可在模型调用前检查已准备的输入，并手动重新引入或移除推理 ID。

#### 何时使用 `'omit'`

当您希望回放的推理条目被规范化为不带 ID（例如简化转发/回放的模型输入，或满足应用流水线中的集成要求）时，使用 `'omit'`。

如果后端/提供方因请求校验错误而拒绝回放的推理条目（例如与后续输入中的推理条目 ID 相关的 HTTP `400` 错误），这也是一个有用的故障排除选项。在这些情况下，使用 `'omit'` 去除回放的推理 ID 可以避免发送被后端视为对新请求无效的 ID。

如果您希望 SDK 在回放的输入中保留推理条目 ID，且您的集成能接收它们，则保持 `'preserve'`。

## 错误与恢复

### 错误处理器

使用 `errorHandlers` 将受支持的运行时错误转换为最终输出而不是抛出。目前仅支持 `maxTurns`。

- `errorHandlers.maxTurns` 仅处理最大轮次错误。
- `errorHandlers.default` 作为受支持类型的回退处理器。
- 处理器接收 `{ error, context, runData }`，并可返回 `{ finalOutput, includeInHistory? }`。

### 异常

SDK 会抛出一小部分可捕获的错误：

- [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)——达到 `maxTurns`。
- [`ModelBehaviorError`](/openai-agents-js/openai/agents-core/classes/modelbehaviorerror)——模型产生了无效输出（如格式错误的 JSON、未知工具）。
- [`InputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/inputguardrailtripwiretriggered) / [`OutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/outputguardrailtripwiretriggered)——护栏违规。
- [`ToolInputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/toolinputguardrailtripwiretriggered) / [`ToolOutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/tooloutputguardrailtripwiretriggered)——工具护栏违规。
- [`GuardrailExecutionError`](/openai-agents-js/openai/agents-core/classes/guardrailexecutionerror)——护栏执行失败。
- [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror)——函数工具超过 `timeoutMs`，且使用了 `timeoutBehavior: 'raise_exception'`。
- [`ToolCallError`](/openai-agents-js/openai/agents-core/classes/toolcallerror)——函数工具在非超时错误下执行失败。
- [`UserError`](/openai-agents-js/openai/agents-core/classes/usererror)——基于配置或用户输入抛出的任意错误。

以上均继承自基础类 `AgentsError`，该类可能提供 `state` 属性以访问当前运行状态。

- 输入护栏只在一次运行的最初用户输入上触发，因此必须用相同的输入/上下文开启新的运行来重试——传入已保存的 `state` 不会重新触发输入护栏。
